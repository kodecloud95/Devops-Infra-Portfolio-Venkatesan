Module 04: Configuration & Secrets

Application Configuration

Decoupling configuration and sensitive data from application code.

1. ConfigMap Management

ConfigMaps allow you to decouple environment-specific configuration from your container images.

Imperative Approach (CLI) 
# Create ConfigMap from literal values
kubectl create configmap my-configdata \
  --from-literal=username=admin \
  --from-literal=password=s3cr3t \
  --from-literal=database_url=jdbc:mysql://localhost:3306/mydb

# Create ConfigMap from a JSON file
kubectl create configmap my-configmap --from-file=my-data.json
Declarative Approach (YAML) 
apiVersion: v1
kind: ConfigMap
metadata:
  name: my-configmap
  labels:
    app: myapplication
data:
  username: "admin"
  password: "s3cr3t"
  database_url: "jdbc:mysql://localhost:3306/mydb"

# Example: Mapping a whole file into a ConfigMap key
data:
  my-data.json: |
    username: "admin"
    password: "s3cr3t"
    database_url: "jdbc:mysql://localhost:3306/mydb"

2. Secret Management

Secrets are used to store sensitive information like TLS certificates, passwords, and SSH keys.

Imperative Secrets (CLI) 
# Generic (Opaque) Secret from literals
kubectl create secret generic my-secret-data \
  --from-literal=username=admin \
  --from-literal=password=s3cr3t

# Create Secret from multiple files
kubectl create secret generic my-secret-data \
  --from-file=username=./username.txt \
  --from-file=password=./password.json
Declarative Secrets (Types & Data) 
apiVersion: v1
kind: Secret
metadata:
  name: my-secret-data
type: Opaque # Default type
data:
  username: YWRtaW4= # base64 of 'admin'
  password: czNjcjN0 # base64 of 's3cr3t'

# Basic Auth Type
type: kubernetes.io/basic-auth
stringData: # Allows non-base64 input (K8s encodes it for you)
  username: admin

# TLS Secret
type: kubernetes.io/tls
data:
  tls.crt: LS0tLS1CRUdJ...
  tls.key: LS0tLS1CRUdJ...

# Docker Registry Secret
type: kubernetes.io/dockerconfigjson
data:
  .dockerconfigjson: ewogICJhdXRocyI6IHsK...

# Service Account Token
type: kubernetes.io/service-account-token
metadata:
  annotations:
    kubernetes.io/service-account.name: my-serviceaccount

3. Injecting Data into Pods

Data can be injected via Environment Variables (individual or bulk) or mounted as Filesystem Volumes.

Deployment Integration Template 
apiVersion: apps/v1
kind: Deployment
spec:
  template:
    spec:
      containers:
      - name: busybox
        image: busybox
        args: ["$Username", "$Password"]
        
        # Method 1: Load ALL keys from ConfigMap/Secret
        envFrom:
        - configMapRef:
            name: my-configmap
        - secretRef:
            name: my-secret

        # Method 2: Load SPECIFIC keys
        env:
        - name: Username
          valueFrom:
            configMapKeyRef:
              name: my-configmap
              key: Username

        # Method 3: Mounting as Volumes (Files)
        volumeMounts:
        - name: config-vol
          mountPath: /etc/config
      
      volumes:
      - name: config-vol
        configMap:
          name: my-configmap
      - name: secret-vol
        secret:
          secretName: my-secret
Pro Tip: Use stringData for Secrets in YAML to avoid manual base64 encoding. Kubernetes will automatically encode the values for you when the object is created.
